# Playwright + Mendix 11 Testing Gotchas

Issues discovered while writing automated Playwright tests against a Mendix 11 app (no authentication, DataGrid 2, NavigationList sidebar). Shared with the mxcli team to improve the `test-app` skill template.

---

## 1. `waitForLoadState('networkidle')` hangs forever

### Problem

The `test-app` skill template includes this in the smoke test example:

```typescript
await page.waitForLoadState('networkidle');
```

This **never resolves**. Mendix maintains a permanent long-polling XHR connection to the server for real-time updates. Playwright's `networkidle` condition requires no network activity for 500ms, which never happens. The test hangs until the global timeout kills it.

### Fix

Replace every `waitForLoadState('networkidle')` with an element-based wait:

```typescript
// ❌ Hangs forever
await page.waitForLoadState('networkidle');

// ✅ wait for a specific element instead
await expect(page.locator('.mx-page')).toBeVisible({ timeout: 15000 });
```

For navigation events, wait for the target page's content widget rather than network state:

```typescript
await page.locator('.mx-name-myNavButton').click();
await expect(page.locator('.mx-name-myDataGrid')).toBeVisible({ timeout: 15000 });
```

### Impact

Any test generated from the current skill template that includes `waitForLoadState('networkidle')` will hang for the full timeout duration (default 30s), making the entire test suite appear to hang.

## 2. NavigationList item selectors have a `text_` prefix in the DOM

### Problem

The skill recommends naming NavigationList items and then selecting them by `mx-name-<itemName>`:

```sql
-- MDL
navigationlist navMain {
  item itemDrivers (action: show_page Module.Drivers_Overview) { ... }
}
```

```typescript
// Skill template selector — does not work
await page.locator('.mx-name-itemDrivers').click();
```

The NavigationList item container (`<li>`) **does not receive a `mx-name-*` class**. Instead, the text span *inside* the item receives a prefixed name:

```html
<!-- Actual DOM -->
<li class="mx-listitem">
  <span class="mx-text mx-name-text_itemDrivers">Drivers</span>
</li>
```

The selector must use the `text_` prefix:

```typescript
// ✅ Correct
await page.locator('.mx-name-text_itemDrivers').click();
```

### Fix

Update the skill template to document that NavigationList item selectors require the `text_` prefix. Alternatively, mxcli could apply the `mx-name-*` class to the `<li>` container element directly for consistency.

### Impact

All NavigationList-based navigation tests fail with "element not found" and time out (30s per test). With 14 entity pages in our test suite, this caused ~7 minutes of hanging before timeouts fired.

## 3. Top navigation bar clicks are intercepted by `div.mx-placeholder`

### Problem

Clicking a top navigation bar item (e.g., a custom nav link added via `create or replace navigation`) fails silently. The element is found and appears clickable, but navigation never fires:

```
error: element intercepted
- locator resolved to <a href="#" role="menuitem" class="mx-name-navigationTree1-1"> Formula 1</a>
- <div class="mx-placeholder">…</div> from <div class="mx-scrollcontainer-center">…</div>
  subtree intercepts pointer events
```

Using `{ force: true }` bypasses Playwright's interception check, but still does not trigger the Mendix navigation handler — the click event fires on the element but the app does not navigate.

### Fix

Use `dispatchEvent('click')` instead. This dispatches the event directly to the element, bypassing the placeholder interception and correctly triggering Mendix's event listeners:

```typescript
// ❌ Fails — pointer intercepted, navigation does not fire
await page.locator('.mx-name-navigationTree1-1').click();

// ❌ Also fails — bypasses check but Mendix handler still not triggered
await page.locator('.mx-name-navigationTree1-1').click({ force: true });

// ✅ Works — dispatches event directly, Mendix navigation fires
await page.locator('.mx-name-navigationTree1-1').dispatchEvent('click');
```

### Additional note on navigation tree numbering

The `navigationTree<N>-<M>` index depends on how many navigation widgets are on the current page — it can differ between the home page and an inner page. The index should be discovered by inspecting the live DOM on the specific page where the click is needed, not assumed to be constant.

### Impact

Navigation to app sections via the top bar appears to work (no error thrown with `force: true`) but the page never changes, causing all downstream assertions to fail with "element not found."

## 4. DataGrid 2 header and data rows both have `role="row"`

### Problem

The skill mentions waiting for DataGrid 2 rows, but does not clarify that both the **header row** and **data rows** share `role="row"`. Selecting `[role="row"].first()` returns the column header row, not the first data row:

```typescript
// ❌ returns the header row: "First Name | Last Name | Nationality | Total Points"
const firstRow = page.locator('.mx-name-myGrid [role="row"]').first();
await expect(firstRow).toContainText('Hamilton'); // fails
```

### Fix

Filter by `role="gridcell"` (present in data rows) vs `role="columnheader"` (present in the header row):

```typescript
// ✅ Targets only data rows, skips the header
const dataRows = page.locator('.mx-name-myGrid [role="row"]:has([role="gridcell"])');
await expect(dataRows.first()).toContainText('Hamilton');
await expect(dataRows.nth(1)).toContainText('Vettel');
```

### Impact

Tests that assert specific data in a DataGrid row will silently match the column headers instead of data, causing confusing assertion failures.
