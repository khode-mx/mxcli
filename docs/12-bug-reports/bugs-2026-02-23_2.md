# mxcli Catalog Bug Report — Session 2026-02-23

## CAT-001: CATALOG.WIDGETS is empty in fast catalog mode

**Severity:** High
**Area:** CATALOG / REFRESH CATALOG

`CATALOG.WIDGETS` returns zero rows when the catalog was built with `refresh catalog` (fast mode). No error or warning is shown — the query succeeds silently with `(no results)`. Widgets are only populated when the catalog is built with `refresh catalog full`.

**Repro:**
```sql
refresh catalog force;
select Name from CATALOG.WIDGETS limit 1;
-- Returns: (no results) — 0 rows
```

**Fix:**
```sql
refresh catalog full;
select Name from CATALOG.WIDGETS limit 1;
-- Returns: 1 result
```

**Expected:** Either:
- `refresh catalog` (fast mode) should populate `CATALOG.WIDGETS`, or
- Querying `CATALOG.WIDGETS` when it was built in fast mode should produce a warning such as: `CATALOG.WIDGETS is not available in fast mode — run refresh catalog full`

## CAT-002: `widgettype` column name conflicts with MDL reserved word

**Severity:** High
**Area:** CATALOG / MDL Parser

The `CATALOG.WIDGETS` table has a column named `widgettype`. However, `widgettype` conflicts with MDL parser reserved words (`widget` + `type`), making it impossible to reference this column by name in a `select` list or `where` clause. The parser produces cascading errors with no useful diagnostic.

**Repro:**
```sql
select widgettype from CATALOG.WIDGETS limit 1;
```

**Error:**
```
Parse error: line 1:7 no viable alternative at input 'SELECTWidgetType'
Parse error: line 1:39 mismatched input 'LIMIT' expecting {GROUP_BY, select, where, having}
```

The same error occurs in a `where` clause:
```sql
select Name from CATALOG.WIDGETS where widgettype like '%Navigation%';
-- Parse error: extraneous input 'WidgetType' expecting ...
```

**Workaround:** Use `select *` to retrieve all columns including `widgettype`, then filter results externally.

**Expected:** Either:
- Column names in catalog queries should be quoted/escaped so reserved words don't conflict, or
- The column should be renamed to something that doesn't collide (e.g., `WidgetKind`, `WidgetClass`)

## CAT-003: Non-existent column `WidgetId` — misleading schema

**Severity:** Low
**Area:** CATALOG.WIDGETS

`CATALOG.WIDGETS` has no `WidgetId` column. The unique identifier column is named `Id` and the type column is `widgettype` (see CAT-002). The naming is inconsistent with other catalog tables (e.g., `CATALOG.ENTITIES` uses `QualifiedName` as its primary identifier, not `Id`), making it easy to guess the wrong column name.

**Repro:**
```sql
select Name, WidgetId from CATALOG.WIDGETS limit 1;
```

**Error:**
```
error: failed to execute catalog query
sql logic error: no such column: WidgetId (1)
```

**Actual columns:** `Id`, `Name`, `widgettype`, `ContainerId`, `ContainerQualifiedName`, `ContainerType`, `ModuleName`, `folder`, `EntityRef`, `AttributeRef`, `description`, ...

**Expected:** Documentation or `HELP catalog` output should list the available columns for each catalog table so users don't have to guess.
