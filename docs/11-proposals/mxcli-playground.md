# Proposal: mxcli Playground

## Overview

A public GitHub repository (`mendixlabs/mxcli-playground`) containing a ready-to-use Mendix project pre-configured with mxcli, Claude Code skills, and a Codespaces devcontainer. Users click "Open in Codespace" and immediately start editing a Mendix app with mxcli -- no local setup required.

## Goals

1. **Zero-friction onboarding** -- clone, open Codespace, start using mxcli in under 2 minutes
2. **Always up-to-date** -- every successful mxcli release automatically updates the playground
3. **Self-contained** -- includes a sample Mendix project with enough structure to be interesting (entities, pages, microflows, security)
4. **Educational** -- includes guided tutorials and example scripts

## Devcontainer

The playground uses the devcontainer generated by `mxcli init` -- no custom configuration to maintain. This already provides:

- JDK 21 (for mxbuild / `mx check`)
- Docker-in-Docker (for `mxcli docker build/check`)
- Claude Code CLI
- mxcli binary download (version-pinned)
- VS Code MDL extension
- mxbuild auto-download for the project's Mendix version

Since the devcontainer is part of what `mxcli init` outputs, it stays current through the same sync pipeline that updates skills and commands. No separate maintenance required.

## Sample Mendix Project

The playground ships with a small but representative Mendix project ("TaskManager") that demonstrates common patterns:

### Domain Model
- **TaskManager.Task** -- String Title, String Description, Enum Status, DateTime DueDate, Boolean Completed
- **TaskManager.Category** -- String Name, String Color
- **TaskManager.Task_Category** -- Association (many-to-one)
- **Administration.Account** -- System user entity (from default template)

### Existing Content
- Basic navigation with home page
- One overview page for Tasks
- One microflow (DS_GetActiveTasks)
- User roles: Admin, User
- Module security configured

This gives users enough to explore (`SHOW STRUCTURE`, `SHOW ENTITIES`, `DESCRIBE`) while leaving room to build on top (add pages, microflows, associations, security rules).

### Creation & Updates
The initial project is created with `mx create-project` and seeded with sample content via mxcli MDL scripts. When a new Mendix version is detected (Pipeline 2), the project is recreated from scratch with the new version. The sample content (entities, pages, microflows) would need to be re-applied -- this can be automated by committing the seed scripts and running them after `mx create-project`.

## Automated Update Pipelines

Two independent workflows keep the playground current: one tracks mxcli releases, the other tracks Mendix releases.

### Pipeline 1: mxcli Release Update

Triggered by a successful mxcli release in the ModelSDKGo repo. Updates tooling only -- does not touch the Mendix project.

```yaml
# .github/workflows/update-playground.yml (in ModelSDKGo repo)
name: Update Playground

on:
  workflow_run:
    workflows: ["Release"]
    types: [completed]

permissions:
  contents: read

jobs:
  update-playground:
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5

      - name: Get release version
        id: version
        run: echo "tag=${GITHUB_REF#refs/tags/}" >> "$GITHUB_OUTPUT"

      - name: Build mxcli
        run: make build

      - name: Generate playground assets
        run: |
          mkdir -p playground-update

          # 1. Version pin
          echo "${{ steps.version.outputs.tag }}" > playground-update/.mxcli-version

          # 2. Run mxcli init in dry-run mode to extract latest assets
          #    (skills, commands, lint rules, AGENTS.md, CLAUDE.md, devcontainer)
          ./bin/mxcli init --output playground-update --tool claude --all-tools --dry-run

          # 3. Copy VS Code extension
          cp vscode-mdl/vscode-mdl-*.vsix playground-update/.devcontainer/vscode-mdl.vsix

      - name: Push to playground repo
        env:
          PLAYGROUND_TOKEN: ${{ secrets.PLAYGROUND_REPO_TOKEN }}
        run: |
          git clone https://x-access-token:${PLAYGROUND_TOKEN}@github.com/mendixlabs/mxcli-playground.git target

          # Sync generated files (preserve app.mpr, scripts, tutorials)
          rsync -av --exclude='app.mpr' --exclude='mprcontents/' \
                    --exclude='scripts/' --exclude='tutorials/' \
                    --exclude='.git' \
                    playground-update/ target/

          cd target
          git config user.name "mxcli-bot"
          git config user.email "mxcli-bot@mendix.com"

          if git diff --quiet && git diff --cached --quiet; then
            echo "No changes to push"
            exit 0
          fi

          git add -A
          git commit -m "chore: update mxcli to ${{ steps.version.outputs.tag }}"
          git push origin main
```

### Pipeline 2: Mendix Version Update

A scheduled workflow in the playground repo that checks for new Mendix releases and recreates the sample project when one is found.

**How it works**: Mendix publishes mxbuild releases on their CDN at `https://cdn.mendix.com/runtime/mxbuild-{version}.tar.gz`. The workflow probes for new patch/minor versions by incrementing from the current version. When a new version is found, it downloads mxbuild, runs `mx create-project` to generate a fresh project, and commits the result.

```yaml
# .github/workflows/update-mendix-version.yml (in mendixlabs/mxcli-playground repo)
name: Update Mendix Version

on:
  schedule:
    - cron: '0 8 * * 1'  # Every Monday at 08:00 UTC
  workflow_dispatch:       # Manual trigger

permissions:
  contents: write

jobs:
  check-mendix-version:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5

      - name: Read current Mendix version
        id: current
        run: echo "version=$(cat .mendix-version)" >> "$GITHUB_OUTPUT"

      - name: Check for newer Mendix versions
        id: check
        run: |
          CURRENT="${{ steps.current.outputs.version }}"
          MAJOR=$(echo "$CURRENT" | cut -d. -f1)
          MINOR=$(echo "$CURRENT" | cut -d. -f2)
          PATCH=$(echo "$CURRENT" | cut -d. -f3)

          # Strategy: probe patch increments first, then minor increments
          LATEST="$CURRENT"

          # Check patch versions (e.g., 10.21.1 → 10.21.2, 10.21.3, ...)
          for i in $(seq 1 20); do
            CANDIDATE="$MAJOR.$MINOR.$((PATCH + i))"
            STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
              "https://cdn.mendix.com/runtime/mxbuild-${CANDIDATE}.tar.gz")
            if [ "$STATUS" = "200" ]; then
              LATEST="$CANDIDATE"
            else
              break
            fi
          done

          # Check next minor version (e.g., 10.21.x → 10.22.0)
          CANDIDATE="$MAJOR.$((MINOR + 1)).0"
          STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
            "https://cdn.mendix.com/runtime/mxbuild-${CANDIDATE}.tar.gz")
          if [ "$STATUS" = "200" ]; then
            # Found new minor -- scan its patches too
            NEW_MINOR=$((MINOR + 1))
            LATEST="$CANDIDATE"
            for i in $(seq 1 20); do
              CANDIDATE="$MAJOR.$NEW_MINOR.$i"
              STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
                "https://cdn.mendix.com/runtime/mxbuild-${CANDIDATE}.tar.gz")
              if [ "$STATUS" = "200" ]; then
                LATEST="$CANDIDATE"
              else
                break
              fi
            done
          fi

          if [ "$LATEST" = "$CURRENT" ]; then
            echo "No new Mendix version found (current: $CURRENT)"
            echo "found=false" >> "$GITHUB_OUTPUT"
          else
            echo "New Mendix version found: $LATEST (was: $CURRENT)"
            echo "found=true" >> "$GITHUB_OUTPUT"
            echo "version=$LATEST" >> "$GITHUB_OUTPUT"
          fi

      - name: Download mxbuild
        if: steps.check.outputs.found == 'true'
        run: |
          VERSION="${{ steps.check.outputs.version }}"
          curl -fsSL "https://cdn.mendix.com/runtime/mxbuild-${VERSION}.tar.gz" \
            -o mxbuild.tar.gz
          mkdir -p mxbuild
          tar xzf mxbuild.tar.gz -C mxbuild

      - name: Create fresh Mendix project
        if: steps.check.outputs.found == 'true'
        run: |
          VERSION="${{ steps.check.outputs.version }}"

          # Remove old project files
          rm -rf app.mpr mprcontents/

          # Create new blank project with the latest Mendix version
          ./mxbuild/modeler/mx create-project app.mpr

          # Update version pin
          echo "$VERSION" > .mendix-version

      - name: Re-run mxcli init
        if: steps.check.outputs.found == 'true'
        run: |
          # Download latest mxcli
          MXCLI_VERSION=$(cat .mxcli-version)
          curl -fsSL "https://github.com/mendix/ModelSDKGo/releases/download/${MXCLI_VERSION}/mxcli-linux-amd64" \
            -o /usr/local/bin/mxcli
          chmod +x /usr/local/bin/mxcli

          # Re-initialize to update devcontainer (which pins mxbuild version from .mpr)
          mxcli init -p app.mpr --tool claude --all-tools --force

      - name: Commit and push
        if: steps.check.outputs.found == 'true'
        run: |
          VERSION="${{ steps.check.outputs.version }}"
          git config user.name "mxcli-bot"
          git config user.email "mxcli-bot@mendix.com"
          git add -A
          git commit -m "chore: update Mendix project to $VERSION"
          git push origin main
```

### Version Files

The playground repo tracks two independent version pins:

| File | Purpose | Updated by |
|------|---------|------------|
| `.mxcli-version` | mxcli binary version (e.g., `v0.42.0`) | Pipeline 1 (mxcli release) |
| `.mendix-version` | Mendix project version (e.g., `10.21.0`) | Pipeline 2 (Mendix version check) |

### What Gets Updated

| Asset | Updated by | Trigger |
|-------|------------|---------|
| `.mxcli-version` | Pipeline 1 | mxcli release |
| Skills, commands, lint rules | Pipeline 1 | mxcli release |
| AGENTS.md, CLAUDE.md | Pipeline 1 | mxcli release |
| VS Code extension | Pipeline 1 | mxcli release |
| Devcontainer | Both | mxcli release or Mendix version change |
| `.mendix-version` | Pipeline 2 | New Mendix version on CDN |
| `app.mpr` / `mprcontents/` | Pipeline 2 | New Mendix version on CDN |

### What Does NOT Get Updated Automatically
- `scripts/` -- hand-written tutorial scripts (may need manual review after major Mendix version bumps)
- `tutorials/` -- hand-written documentation
- `README.md` -- manual updates only

## User Experience

### First-Time User Flow

1. Visit `github.com/mendixlabs/mxcli-playground`
2. Click **"Open in Codespace"** (or `gh codespace create`)
3. Wait ~90 seconds for container build + mxcli download
4. Terminal opens with welcome message:
   ```
   Welcome to the mxcli Playground!

   Quick start:
     mxcli -p app.mpr              # Open interactive REPL
     mxcli exec scripts/01-explore.mdl -p app.mpr   # Run example script

   Or use Claude Code:
     claude                         # Start Claude Code
     /explore                       # Explore the project
     /create-entity                 # Create a new entity

   See tutorials/ for step-by-step guides.
   ```
5. User can immediately run `mxcli` or launch Claude Code with full skill support

### Claude Code Experience

With the pre-configured `.claude/` directory, users get:
- All slash commands working out of the box (`/explore`, `/create-entity`, `/create-crud`, `/lint`, etc.)
- Skills loaded for microflow, page, and domain model authoring
- Lint rules for quality checks
- MDL syntax highlighting via the VS Code extension

### Non-Claude Users

The `.ai-context/skills/` and `AGENTS.md` files support Cursor, Windsurf, Continue.dev, and Aider users with the same skill content.

## mxcli init --dry-run Flag

To support the automated pipeline, add a `--dry-run` flag to `mxcli init` that writes all generated assets to an `--output` directory without requiring an actual `.mpr` file or modifying a real project. This is a small addition to `cmd/mxcli/init.go`.

Alternatively, the workflow could use a temp `.mpr` file and run `mxcli init` normally, then cherry-pick the generated files.

## Open Questions

1. **Public vs private repo?** -- Public maximizes reach; private limits to Mendix community members
2. **mxbuild in container?** -- The setup script downloads mxbuild (~500MB). Should we pre-bake it into the Docker image for faster startup at the cost of larger image size?
3. **Multiple sample projects?** -- Start with one (TaskManager), but could add more later (e.g., a blank project, a complex reference app)
4. **Prebuilds** -- Enable Codespaces prebuilds to reduce startup time from ~90s to ~15s (requires org-level config)
5. **Seed script resilience** -- When Mendix version updates recreate the project, the seed MDL scripts must still work. Should Pipeline 2 run `mxcli check` on the scripts before committing, and open an issue if they fail?
6. **Major version jumps** -- The CDN probe scans patch and next-minor. For major version bumps (e.g., 10.x → 11.0.0), a manual trigger or expanded probe range is needed

## Alternatives Considered

### 1. Codespace on the main ModelSDKGo repo
- **Pro**: No separate repo to maintain
- **Con**: Includes full SDK dev environment (Go, ANTLR, JDK 17) -- heavy, slow, confusing for end users who just want to use mxcli

### 2. GitHub template repository
- **Pro**: Users get their own copy to experiment with
- **Con**: Harder to keep up-to-date; users must re-create from template to get updates

### 3. Docker image on Docker Hub
- **Pro**: Works outside GitHub Codespaces
- **Con**: More friction (install Docker, pull image, mount project); misses VS Code extension and Claude Code integration

**Recommendation**: Primary approach is the Codespace playground repo. Consider additionally publishing a Docker image for non-Codespace users as a follow-up.
