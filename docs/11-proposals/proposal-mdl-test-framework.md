# Proposal: MDL Test Framework

**Status:** Draft
**Date:** 2026-02-23
**Author:** AI-assisted design

---

## Summary

A testing framework that uses MDL (Mendix Definition Language) as the test definition language,
with javadoc-style annotations for expectations and verification. Tests live outside the project
and are sent to the runtime for execution, keeping the project under test clean.

## Motivation

Testing microflows generated by MDL scripts currently requires a manual process:
1. Create a TestRunner microflow
2. Set it as after-startup
3. Build and restart the runtime
4. Manually inspect logs for pass/fail

This is tedious, not automatable in CI, and produces no machine-readable results.
We need a framework that:

- Uses a **familiar syntax** (MDL itself) for test definitions
- Supports **self-documenting tests** via javadoc annotations
- Keeps tests **outside the project** (not polluting the app model)
- Produces **standard output** (JUnit XML) for CI integration
- Can evolve from simple log-based execution to **HTTP-based runtime execution**

## Design

### Test Definition Language

Tests are written in MDL with javadoc annotations that describe expectations:

```sql
/**
 * @test String concatenation
 * @expect $result = 'John Doe'
 */
$result = call microflow MfTest.M003_StringOperations(
  FirstName = 'John', LastName = 'Doe'
);
/

/**
 * @test Entity creation and persistence
 * @expect $product/Name = 'TestProduct'
 * @expect $product/Code = 'TP-001'
 * @verify SELECT count(*) FROM MfTest.Product WHERE Code = 'TP-001' = 1
 */
$product = call microflow MfTest.M012_CreateEntity(
  Name = 'TestProduct', Code = 'TP-001'
);
commit $product;
/

/**
 * @test Nested IF returns correct grade
 * @expect $result = 'B'
 */
$result = call microflow MfTest.M008_NestedIf(Score = 85);
/

/**
 * @test Error handling on invalid input
 * @throws 'Validation failed'
 */
call microflow MfTest.M050_ValidateInput(value = -1);
/
```

Each test block is a sequence of MDL microflow body statements, preceded by a javadoc
comment containing test annotations. Blocks are separated by `/` (the MDL statement terminator
used for microflows).

### Annotation Tags

| Tag | Purpose | Example |
|-----|---------|---------|
| `@test` | Test name (required) | `@test string concatenation` |
| `@expect` | Assert variable value | `@expect $result = 'John Doe'` |
| `@expect` | Assert attribute on entity | `@expect $product/Name = 'TestProduct'` |
| `@verify` | OQL post-condition | `@verify select count(*) from Mod.E where Code = 'X' = 1` |
| `@setup` | Reference to setup block | `@setup createTestData` |
| `@cleanup` | Rollback strategy | `@cleanup rollback` (default) or `@cleanup none` |
| `@throws` | Expect error message | `@throws 'validation failed'` |

### Two File Formats

**1. Pure MDL test files** (`.test.mdl`):

Test blocks separated by `/`, each with javadoc annotations. Simple, minimal, easy to write.

**2. Markdown specification documents** (`.test.md`):

Literate testing — tests embedded in documentation using `` ```mdl-test `` fenced code blocks.
The surrounding markdown serves as human-readable specification. This makes the test file
also serve as living documentation.

~~~markdown
# MfTest Specification

## string Operations

The M003 microflow concatenates first and last names with a space separator.

```mdl-test
/** @expect $result = 'John Doe' */
$result = call microflow MfTest.M003_StringOperations(
  FirstName = 'John', LastName = 'Doe'
);
```

## Arithmetic

Division should return a decimal result.

```mdl-test
/** @expect $result = 25.0 */
$result = call microflow MfTest.M010_ArithmeticOperations(
  Total = 100.0, count = 4
);
```
~~~

### Architecture

```
┌─────────────────┐     ┌──────────────────────┐     ┌─────────────────────┐
│  .test.mdl      │     │      mxcli test       │     │   Mendix runtime    │
│  .test.md       │────▶│                        │────▶│                     │
│                 │     │  1. Parse test file    │     │  Phase 1:           │
│  test specs     │     │  2. Extract test blocks│     │  after-startup MF   │
│  with @expect   │     │  3. generate runner MF │     │  (log-based)        │
│                 │     │  4. build & restart    │     │                     │
│                 │     │  5. Parse logs         │     │  Phase 2:           │
│                 │     │  6. Output JUnit xml   │     │  /test/ endpoint    │
│                 │     │                        │     │  (HTTP-based)       │
└─────────────────┘     └──────────────────────┘     └─────────────────────┘
```

### Test Execution Flow (Phase 1: After-Startup)

1. **Parse** the test file(s) to extract test blocks with annotations
2. **Generate** a `MxTest.TestRunner` microflow that:
   - Wraps each test in a LOG+CALL+LOG pattern
   - Outputs structured log lines: `MXTEST:PASS:testName` or `MXTEST:FAIL:testName:message`
   - Validates `@expect` assertions using IF checks on return values
   - Returns Boolean (required by Mendix for after-startup)
3. **Inject** the TestRunner into the project (CREATE OR REPLACE)
4. **Set** `AfterStartupMicroflow = 'MxTest.TestRunner'`
5. **Build** and restart the runtime (`mxcli docker build` + `docker down` + `docker up`)
6. **Parse** runtime logs for `MXTEST:` lines
7. **Verify** `@verify` OQL assertions via M2EE `preview_execute_oql`
8. **Cleanup**: remove TestRunner, restore original after-startup setting
9. **Output** results as JUnit XML, TAP, or console table

### Generated TestRunner Pattern

For a test file with two test blocks, the generated MDL looks like:

```sql
create or replace microflow MxTest.TestRunner ()
returns boolean as $success
begin
  declare $success boolean = true;
  declare $TestFailed boolean = false;

  log info node 'MXTEST' 'MXTEST:START:suite';

  -- Test 1: String concatenation
  log info node 'MXTEST' 'MXTEST:RUN:test_1:String concatenation';
  set $TestFailed = false;
  $result = call microflow MfTest.M003_StringOperations(
    FirstName = 'John', LastName = 'Doe'
  ) on error {
    log error node 'MXTEST' 'MXTEST:FAIL:test_1:Exception: ' + $latestError;
    set $TestFailed = true;
    set $success = false;
  };
  if not $TestFailed then
    if $result = 'John Doe' then
      log info node 'MXTEST' 'MXTEST:PASS:test_1';
    else
      log error node 'MXTEST' 'MXTEST:FAIL:test_1:Expected $result = ''John Doe'' but got ' + $result;
      set $success = false;
    end if;
  end if;

  -- Test 2: Nested IF returns correct grade
  log info node 'MXTEST' 'MXTEST:RUN:test_2:Nested IF returns correct grade';
  set $TestFailed = false;
  $result2 = call microflow MfTest.M008_NestedIf(
    Score = 85
  ) on error {
    log error node 'MXTEST' 'MXTEST:FAIL:test_2:Exception: ' + $latestError;
    set $TestFailed = true;
    set $success = false;
  };
  if not $TestFailed then
    if $result2 = 'B' then
      log info node 'MXTEST' 'MXTEST:PASS:test_2';
    else
      log error node 'MXTEST' 'MXTEST:FAIL:test_2:Expected $result2 = ''B'' but got ' + $result2;
      set $success = false;
    end if;
  end if;

  log info node 'MXTEST' 'MXTEST:END:suite';
  return $success;
end;
/
```

### Log Protocol

The TestRunner outputs structured log lines that `mxcli test` parses:

| Log Line | Meaning |
|----------|---------|
| `MXTEST:START:suite` | Test suite starting |
| `MXTEST:run:id:name` | Test starting |
| `MXTEST:PASS:id` | Test passed |
| `MXTEST:FAIL:id:message` | Test failed with message |
| `MXTEST:SKIP:id:reason` | Test skipped |
| `MXTEST:end:suite` | Test suite complete |

### JUnit XML Output

```xml
<?xml version="1.0" encoding="UTF-8"?>
<testsuites>
  <testsuite name="MfTest" tests="3" failures="1" errors="0" skipped="0" time="2.45">
    <testcase name="string concatenation" classname="MfTest" time="0.12"/>
    <testcase name="Nested if returns correct grade" classname="MfTest" time="0.08"/>
    <testcase name="entity creation" classname="MfTest" time="0.25">
      <failure message="Expected $product/Name = 'TestProduct' but got 'Other'">
        Expected $product/Name = 'TestProduct' but got 'Other'

</testsuites>
```

### CLI Interface

```bash
# run tests from a .test.mdl file
mxcli test tests/microflows.test.mdl -p app.mpr

# run tests from a markdown spec
mxcli test specs/microflow-spec.test.md -p app.mpr

# run all test files in a directory
mxcli test tests/ -p app.mpr

# Output JUnit xml for CI
mxcli test tests/ -p app.mpr --junit results.xml

# list tests without executing
mxcli test tests/ --list

# Verbose output (show all log lines)
mxcli test tests/ -p app.mpr --verbose

# Skip build (reuse existing deployment)
mxcli test tests/ -p app.mpr --skip-build

# Custom timeout for runtime startup
mxcli test tests/ -p app.mpr --timeout 10m

# run specific test(s) by name
mxcli test tests/ -p app.mpr --run "String concatenation"
mxcli test tests/ -p app.mpr --run "String concatenation" --run "Boolean logic"
```

## Phase 2: HTTP-Based Execution (Future)

Phase 2 replaces the after-startup pattern with a persistent HTTP endpoint in the runtime,
enabling test execution without restarts.

### MxTest Java Module

A Mendix module containing:
- A **Java action** registered as a `RequestHandler` at `/mxtest/` in after-startup
- An **AST interpreter** that executes test steps using Mendix Core API

The Java interpreter maps AST nodes to Core API calls:

| AST Step | Java Core API |
|----------|---------------|
| `callMicroflow` | `Core.microflowCall(name).withParam(...).execute(ctx)` |
| `createObject` | `Core.instantiate(ctx, "Module.Entity")` + `setValue()` |
| `changeObject` | `obj.setValue(ctx, "attr", value)` |
| `commit` | `Core.commit(ctx, obj)` |
| `delete` | `Core.delete(ctx, obj)` |
| `retrieve` | `Core.retrieveXPathQuery(ctx, xpath)` |
| `declareVariable` | Store in local `map<string, object>` |

### Wire Format (JSON AST)

mxcli parses the MDL test file and sends a JSON AST to the `/mxtest/` endpoint:

```json
{
  "tests": [
    {
      "id": "test_1",
      "name": "string concatenation",
      "steps": [
        {
          "type": "callMicroflow",
          "target": "$result",
          "microflow": "MfTest.M003_StringOperations",
          "params": {
            "FirstName": {"type": "string", "value": "John"},
            "LastName": {"type": "string", "value": "Doe"}
          }
        }
      ],
      "expect": [
        {"variable": "$result", "operator": "=", "value": "John Doe"}
      ],
      "verify": [],
      "cleanup": "rollback"
    }
  ]
}
```

### Transaction Isolation

Each test runs in its own transaction context:
- **Default (`@cleanup rollback`)**: Transaction is rolled back after assertions
- **`@cleanup none`**: Transaction is committed (for tests that verify persistence via `@verify`)

## Phase 4: VS Code IDE Integration

### Executable Markdown Tests

The `.test.md` format creates a **notebook-like experience** in VS Code where documentation
and executable tests live side by side. Each `mdl-test` code block shows clickable run buttons,
turning the specification document into an interactive test runner — similar to how data
scientists work with Jupyter notebooks.

### Phase 4a: CodeLens Provider

When opening a `.test.md` file in VS Code with the MDL extension installed:

1. A **"Run All Tests (N tests)"** CodeLens button appears at the top of the file
2. Each `` ```mdl-test `` code block shows a **"Run Test: {name}"** CodeLens above it
3. Clicking a CodeLens runs the test in a VS Code terminal with colored output
4. Results appear inline in the terminal (pass/fail with colors)

```
  ┌─────────────────────────────────────────────────────────────┐
  │  microflow-spec.test.md                                     │
  ├─────────────────────────────────────────────────────────────┤
  │  ▶ run all Tests (15 tests)                                 │  ← CodeLens
  │                                                             │
  │  # MfTest Specification                                     │
  │                                                             │
  │  ## string Operations                                       │
  │                                                             │
  │  The M003 microflow concatenates first and last names.      │
  │                                                             │
  │  ▶ run Test: string concatenation                           │  ← CodeLens
  │  ┌─────────────────────────────────────────────────────┐    │
  │  │ /** @test string concatenation                      │    │
  │  │  * @expect $result = 'John Doe' */                  │    │
  │  │ $result = call microflow MfTest.M003(               │    │
  │  │   FirstName = 'John', LastName = 'Doe'              │    │
  │  │ );                                                  │    │
  │  └─────────────────────────────────────────────────────┘    │
  │                                                             │
  │  ## Arithmetic                                              │
  │                                                             │
  │  ▶ run Test: Division arithmetic                            │  ← CodeLens
  │  ┌─────────────────────────────────────────────────────┐    │
  │  │ /** @test Division arithmetic                       │    │
  │  │  * @expect $result = 25.0 */                        │    │
  │  │ $result = call microflow MfTest.M010(               │    │
  │  │   Total = 100.0, count = 4                          │    │
  │  │ );                                                  │    │
  │  └─────────────────────────────────────────────────────┘    │
  └─────────────────────────────────────────────────────────────┘
```

**Implementation**: A `CodeLensProvider` (~150 lines TypeScript) scans the markdown for
`` ```mdl-test `` fenced blocks, extracts `@test` names, and creates CodeLens items that
invoke `mxcli test <file> --run "<name>"` via the existing terminal execution pattern.

**CLI requirement**: The `--run` flag on `mxcli test` filters execution to specific test(s)
by name, so only the selected test is included in the generated TestRunner microflow.

**Tradeoff**: Each test execution triggers a Docker build + restart (~30-60 seconds). This is
acceptable for individual test debugging but not for rapid iteration. Phase 2 (HTTP endpoint)
eliminates this rebuild overhead entirely.

### Phase 4b: Test Explorer (TestController API)

The VS Code Testing API provides a richer experience:

- **Test Explorer sidebar** with a tree view of all tests discovered from `.test.md` files
- **Run/debug buttons** per test and per file in the sidebar
- **Pass/fail decorations** inline in the editor (green checkmarks / red X marks)
- **Test result history** across runs

The Test Explorer discovers tests by parsing `mxcli test --list` output and reports results
by parsing JUnit XML output. This builds on the CodeLens foundation and adds persistent
test state tracking.

### Phase 4c: Custom Notebook Controller

A full Notebook API integration where each `mdl-test` block becomes an executable cell with:

- Per-cell output display (pass/fail result rendered below each cell)
- Progress indicators during execution
- Cell-by-cell execution with play buttons
- Closest experience to Jupyter-style data science notebooks

This is the most complex option and should only be pursued if the notebook UX is strongly
preferred over CodeLens + Test Explorer. Requires a custom `NotebookSerializer` to parse
`.test.md` into notebook cells and a `NotebookController` to execute them.

### Phased Rollout

| Phase | Approach | Effort | Prerequisite |
|-------|----------|--------|--------------|
| 4a | CodeLens buttons | ~200 lines TS | `--run` CLI flag |
| 4b | Test Explorer sidebar | ~400 lines TS | Phase 4a |
| 4c | Custom Notebook | ~800 lines TS | Phase 4b (optional) |

## Phase 5: Watch Mode

```bash
mxcli test tests/ -p app.mpr --watch
```

Re-runs tests on file changes. Useful during development — modify a microflow, save,
tests re-run automatically. Requires Phase 2 (HTTP endpoint) to avoid restart overhead.

When combined with Phase 4 (VS Code integration), watch mode can auto-update CodeLens
decorations and Test Explorer results as tests re-run in the background.

### Test Annotation Parsing

Javadoc comments are parsed for annotation tags:

```go
type TestAnnotation struct {
    Test    string   // @test name
    Expects []Expect // @expect assertions
    Verify  []string // @verify OQL queries
    Setup   string   // @setup block reference
    Cleanup string   // @cleanup strategy (default: "rollback")
    Throws  string   // @throws expected error message
}

type Expect struct {
    Variable string // $var or $var/attr
    Operator string // = or <>
    value    string // Expected value (string representation)
}
```

### Success Criteria

1. `mxcli test file.test.mdl -p app.mpr` executes all test blocks
2. Console output shows pass/fail per test with colors
3. `--junit results.xml` produces valid JUnit XML
4. `--list` shows test names without executing
5. Tests run in MxTest module context (not polluting app modules)
6. After-startup and security settings are restored after test run
